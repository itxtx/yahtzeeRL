# Code Audit — yahtzeeRL

Scope: all of `yahtzee_rl/` plus tests and the Colab notebook. Focus on implementation correctness first, then an evaluation of the RL approach. All 18 tests pass on CPU (jax 0.6.x, mctx 0.0.5). I also ran independent probes: the env legal mask and the training-time `legal_mask_from_obs` agree on thousands of random mid-game states; self-play returns are exactly antisymmetric between players within each game; MCTS action weights sum to 1 on valid frames with zero mass on illegal actions; games complete within the fixed 78-step scan (~72 typical).

## What is correct

The core game logic is sound. Scoring handles all 13 categories correctly, including the small/large straight window checks, full house requiring exactly 3+2 (a five-of-a-kind scores 0 there, consistent with the documented "no joker rules" simplification), and the 63/35 upper bonus. The env's `step` is a well-constructed branchless batched transition: out-of-range and illegal actions are guarded into no-ops with -1 reward before any gather can clamp them into valid moves, terminal states absorb via a legal no-op action 0, and turn passing, reroll counting, and the next player's opening roll are all handled correctly.

The two-player negamax wiring in `mcts.make_recurrent_fn` is the subtlest part of the codebase and it is implemented correctly: discount is -1 only when the active player changes (hold actions keep +1, so a player's multi-action turn is valued consistently), discount is 0 on transitions into terminal states so the network's value at terminal nodes never leaks into backups, and the terminal reward is computed from the parent's acting player so the perspective matches the edge being backed up. The repeated terminal reward on no-op expansions beyond a terminal node is also harmless precisely because the incoming discount is already 0 — this is the standard mctx absorbing-state pattern done right.

Training targets are consistent end to end: the value head is tanh-bounded, the shaped terminal reward is constructed to stay in [-1, 1], every frame's return is from that frame's acting player's perspective, and done-padding frames are excluded from the loss. Using -1e9 rather than -inf in `masked_logits` avoids the classic 0 × -inf NaN when multiplying MCTS weights by log-probs of illegal actions. The evaluator alternates seats to cancel first-mover advantage, and checkpoint save/restore handles orbax async finalization, temp dirs, numeric step ordering, and cross-device restore correctly.

## Findings

### 1. MCTS determinizes a stochastic game (high impact, the main issue)

`gumbel_muzero_policy` assumes deterministic transitions. Your `recurrent_fn` samples dice with the provided rng, so each tree edge is expanded once and caches a single sampled dice outcome; every subsequent visit to that child reuses that one realization. The Q-value of "reroll all" is therefore the value of one specific roll out of 7776, not an expectation. Verified by probe: the child of the reroll-all edge holds exactly one dice sample per search.

Consequences: reroll decisions — the heart of Yahtzee skill — are valued with single-sample noise; the argmax over noisy Qs is biased toward lucky samples; and the policy targets distilled into the network inherit this noise. The network prior averages some of it out over many games, but the search adds far less than its compute cost suggests and can actively mislead (e.g. a reroll edge that happened to sample a Yahtzee looks better than keeping a made full house).

Fixes, in increasing order of fidelity: (a) the mctx version you depend on already ships `mctx.stochastic_muzero_policy` — restructure the transition into decision nodes (hold/score) and chance nodes (dice outcomes) and let the search marginalize properly; (b) restructure around afterstates with exact expectation — rerolling k dice has only C(k+5, 5) ≤ 252 distinct outcomes with multinomial weights, which is trivially enumerable in JAX, so a depth-limited expectimax with the value net at leaves would be both exact over chance and cheaper than 32 sampled sims; (c) cheapest patch, keep Gumbel MuZero but average several sampled children per chance-heavy edge.

### 2. Unordered dice waste capacity and split search statistics (medium)

Dice are exchangeable but the env keeps them positional. The network sees [5,2,5,1,3] and [1,2,3,5,5] as different states (up to ~120× state inflation for the one-hot encoding), and equivalent hold actions are distinct action ids — keeping the first 5 versus the second 5 of a pair are different actions whose MCTS visit counts and policy mass get split. Sorting dice after every roll inside `env.step`/`reset` is game-equivalent and collapses both problems at once (note it must happen in the env, not just the observation, since hold masks are positional). A canonical hold encoding ("keep m copies of face f") would go further, but sorting alone is one line and pays for itself immediately.

### 3. One gradient step per batch of fresh games (medium, efficiency)

Each update generates 64 full games (≈78 steps × 32 simulations × a network eval each — on the order of 10⁵ forward passes) and then takes a single Adam step over the ~4.6k resulting frames. AlphaZero-style training amortizes generation cost through a replay buffer with many minibatch updates per game generated. A small FIFO buffer of recent trajectories with 4–8 minibatch steps per generation would likely give several times the learning per GPU-hour with no algorithmic risk. Relatedly, 1000 updates × 64 games is a small corpus by AlphaZero standards; expectations should be calibrated accordingly.

### 4. Pure Monte Carlo value targets (low-medium)

Every frame regresses to the final shaped outcome. That is standard AlphaZero, but in a 78-step stochastic game the variance is large. Mixing the target with the search's improved root value (available from the gumbel policy output) or an n-step bootstrap is a low-risk variance reduction. The fixed `gumbel_scale=1.0` with no schedule is also worth revisiting once the policy sharpens.

### 5. Smaller observations

The entropy bonus on a loss that already distills soft MCTS weights is redundant; at 1e-3 it is harmless but can be dropped. The observation normalizes all category scores by 50, which makes upper-bonus arithmetic (thresholds at 63) something the net must reconstruct from six small floats; an explicit "upper total so far / bonus secured" feature is cheap and targets a known discontinuity in the value function. `load_checkpoint_config`'s fallback `TrainConfig(batch_size=1)` silently assumes the default hidden_dim when config.json is missing, which will fail with opaque shape errors for non-default checkpoints — better to raise. In `evaluate.run_games` both policies run every step (an MCTS-vs-MCTS eval pays double search cost) and the heuristic emits illegal actions at terminal states — both harmless, the env guards them, but the second relies on the guard rather than the mask. Test coverage is good for scoring/env/rewards; the negamax sign convention in the recurrent_fn — the most fragile invariant in the codebase — has no test, and a two-ply construction where the right move only looks right after the perspective flip would lock it in.

## Assessment of the RL approach

The architecture — AlphaZero-style self-play with a policy/value net, Gumbel MuZero search over the real env, terminal win/loss reward with bounded margin shaping — is a coherent and well-executed design, and the JAX implementation (everything jitted end to end, search inside `lax.scan`) is genuinely good. The margin-shaped reward is thoughtfully constructed: bounded to match the tanh head, monotone in margin, and it keeps the win/loss signal dominant. Including the opponent's scorecard in the observation is the right call, since optimal two-player Yahtzee is genuinely score-dependent (risk-seeking when behind).

The strategic mismatch is that this machinery is built for adversarial deterministic games, while two-player Yahtzee is nearly a parallel solitaire race with heavy chance. Two consequences follow. First, the deterministic-search assumption is violated (finding 1) — this is the highest-leverage thing to fix, and mctx's stochastic policy or an exact-expectation afterstate search are both within reach of the current code structure. Second, the win/loss objective makes the learning signal sparse and relative when a dense, absolute one is available: single-player expected-score Yahtzee is a well-studied benchmark (optimal play averages ≈254.6), and pretraining the value/policy on solo score maximization before fine-tuning on head-to-head win probability would give the net most of its competence from a much cleaner signal, while also providing an absolute yardstick — mean solo score — to track progress, which the current win-rate-vs-heuristic metric cannot give once the heuristic is saturated.

In short: implementation correctness is strong — I found no outright bugs in game rules, env transitions, perspective handling, or training targets. The gap is methodological: handle chance explicitly in the search, canonicalize dice, and reuse self-play data, roughly in that order of payoff.

## Addendum — fixes implemented

All findings above have since been addressed in this repo. The search now uses `mctx.stochastic_muzero_policy` with explicit afterstates and analytic chance distributions over the 252 sorted dice hands (`dice_tables.py`, rewritten `mcts.py`), with the negamax perspective flip carried on chance edges and exact terminal values substituted at terminal afterstates. The env sorts dice after every roll (rerolls are combined before sorting so rerolled values stay iid). The observation gained upper-bonus progress features and a permutation-invariant dice-count encoding that also represents afterstates (unknown dice + opponent-to-move flag). Training samples minibatches from a replay buffer with several SGD steps per generation, and value targets blend the terminal outcome with the search root value. `load_checkpoint_config` now fails loudly when config.json is missing, evaluation/CLI play run noise-free greedy search (and the evaluator jits per-policy action functions), and the negamax sign convention, chance distributions, sorted-dice invariant, replay buffer, and blended targets are covered by tests (28 passing). Note: checkpoints from before these changes are incompatible — the observation encoding changed, so old params will not load.
